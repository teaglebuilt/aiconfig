  6. Updates docs — if major changes found, updates context/knowledge/docs/ with latest info

  features, and a full run history so you can see the cadence of upstream changes over time.

  Key design decisions:

  - Awareness, not action — the command reports gaps but doesn't refactor aiconfig. You decide what to act
  on.
  - Run history — each run appends to runs[], so over time you'll see patterns (e.g., "Cursor ships weekly,
   Claude Code ships biweekly")
  - Scoped to what matters — focuses on features relevant to aiconfig's mission: hooks, skills, agents,
  MCP, context systems
  - Doc updates gated on significance — only triggers doc scraping if a major version change is detected

  Run it with /vibe_check and it'll do the first baseline check, recording everything for future
  comparisons.

⏺ The workflow is complete. Here's how the full loop works:
                                                                                                         
  Automated Detection (GitHub Actions)
                                                                                                           
  - Runs monthly on the 1st (or manually via workflow_dispatch)                                            
  - Two parallel jobs check Claude Code (via GitHub API) and Cursor (via download manifest) for new        
  versions                                                                                                 
  - Compares against what's stored in memory/global/vibe_check.json                                      
  - If either client has a new release, creates a GitHub issue with a report and links to changelogs       
  - The issue tells you to run /vibe_check locally

  Manual Analysis (Local)

  - You see the issue, open Claude Code, run /vibe_check
  - It does the deep analysis — fetches docs, diffs capabilities, identifies gaps
  - Updates vibe_check.json with new versions and timestamps
  - Next time the workflow runs, it'll compare against the updated state

  The Loop

  GitHub Actions (monthly)  →  detects new release  →  creates issue
           ↑                                                  ↓
    reads vibe_check.json                            you run /vibe_check locally
           ↑                                                  ↓
           └──────────  updates vibe_check.json  ←────────────┘

  Note: You'll want to create the vibe-check label in your repo so the issues get tagged properly. The
  Cursor version detection has a fallback chain since they don't have a public releases API — you may need
  to adjust the endpoint if neither resolves cleanly on first run.
