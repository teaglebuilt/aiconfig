     Summary of the Review Request Architecture v2.0                                                                                                                                                   
                                                                                                                                                                                                       
     The ADR proposes a redesign organized around three pillars:
                                                                                                                                  
     Pillar 1: Unified Fulfillment Pipeline                                                                                                                      

     - Introduces a fulfillment type adapter pattern where each review type (photo/text, live video, video call, future types) implements a common FulfillmentTypeAdapter interface for artifact
     collection, validation, approval rules, and UI rendering.
     - Creates a new fulfillment_artifacts table that normalizes all review artifacts (photos, recordings, notes, transcripts) regardless of source.
     - Eliminates the split between review_fulfillment_drafts and review_fulfillments -- drafts become review_fulfillments with status = 'draft' from the start.
     - Brings Zoom video consultations into the formal fulfillment pipeline for the first time (previously had no approval flow).
     - Adds fulfillment versioning so revisions create new versions with audit trail.

     Pillar 2: LiveKit-Native Real-Time Layer

     - Removes ~2,500 lines of dead code from the video package (unused hooks, components, and provider that were bypassed by the web app using @livekit/components-react directly).
     - Introduces a LiveKit Agent (Node.js, deployed to LiveKit Cloud) that joins live review rooms as an invisible participant to handle: quality monitoring (5s sampling), chat persistence to
     database, 60-second reconnection timeout enforcement, max/min duration tracking, and event emission.
     - Plans mobile live review as a fresh implementation using @livekit/react-native directly, sharing only DAL hooks (platform-agnostic React Query) across web and mobile.

     Pillar 3: Event-Driven Orchestration

     - Introduces a domain_events table with PostgreSQL triggers that emit events on status changes across all core tables (requests, applications, fulfillments, live sessions, payments).
     - Uses pg_notify + Supabase Realtime to route events to processor Edge Functions for notifications, payment triggers, analytics, and scheduling.
     - Adds Upstash Redis for rate limiting (1 active session per reviewer), token caching, session presence, and webhook idempotency.
     - Adds live_review_chat_messages table for persistent chat (no longer deferred to Phase 2).

     Migration Plan

     A 14-week phased rollout: Foundation (weeks 1-3) -> Event Bus (3-4) -> LiveKit Agent (4-6) -> Unified Fulfillment UI (6-8) -> Mobile Live Review (8-12) -> Polish (12-14).

     ADR-0003 redefines messaging as a review-request-scoped engagement thread (Fiverr model) rather than a standalone communication feature. Key architectural decisions:

     1. Single conversation per application -- The conversations table gains NOT NULL constraints on request_id and application_id, with a unique constraint on (application_id). One thread per
     engagement.
     2. System messages for lifecycle events -- The messages table is extended with sender_role, message_type, system_event_type, and system_event_data columns. A database trigger on domain_events
     (from ADR-0002's event bus) automatically inserts system messages into the appropriate conversation when lifecycle events occur (application accepted, fulfillment submitted, payment released,
     live session started/ended, etc.).
     3. Eliminated live_review_chat_messages -- The separate live review chat table from ADR-0002 is removed. The LiveKit Agent's ChatPersister writes directly to the unified messages table, giving
     pre-stream coordination, in-stream chat, and post-stream follow-up a single chronological thread.
     4. Conversation context card -- Each thread displays a collapsible card with request details, current phase, and phase-appropriate actions (view draft, join stream, review recording, etc.).
     5. Automatic conversation creation -- Conversations are created on application.accepted events with the first system message. Pre-acceptance messaging is also supported (Fiverr inquiry model).
